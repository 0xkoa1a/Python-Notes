# C / Rust extension

当 Python 层已经不是合适的执行位置时，可以把核心路径移到 C、C++、Rust 或专门编译工具中。

**扩展语言的目标不是替代 Python，而是把少数性能关键、边界清楚的核心实现为高效组件，让 Python 继续负责组织和调度。**

这是最后几步优化之一，不是默认起手式。

## 1. 什么时候考虑扩展

考虑条件：

- profiling 已确认热点。
- 算法和数据结构已经合理。
- NumPy/PyTorch/标准库无法表达。
- 纯 Python 循环是核心瓶颈。
- 性能收益能抵消构建和维护成本。

不要为了“更底层”而写扩展。扩展会增加：

- 构建复杂度。
- 平台兼容问题。
- 调试成本。
- 内存安全边界。
- 发布成本。

## 2. 可选路线

常见选择：

- C extension：最底层，最贴近 CPython API。
- Cython：Python-like 语法加类型，适合逐步优化。
- pybind11：常用于 C++ 绑定。
- Rust + PyO3 / maturin：用 Rust 写扩展并打包。
- Numba：JIT 编译数值函数。

选择依据不是潮流，而是团队能力、部署环境和问题类型。

## 3. 边界设计

扩展函数应该有清楚边界：

```python
def fast_score(values: list[float]) -> float:
    ...
```

更高效的边界通常使用连续内存数据：

```python
def fast_score_array(values: np.ndarray) -> float:
    ...
```

如果每次调用扩展只处理一个很小对象，跨语言调用开销可能抵消收益。尽量批量传入数据。

## 4. GIL 与扩展

扩展代码可以在长时间计算时释放 GIL，让其他 Python 线程继续运行。

但这要求扩展代码在释放 GIL 期间不操作 Python 对象。它通常只能处理底层内存、数值缓冲区或自己管理的数据结构。

原则：

```text
操作 Python 对象 -> 需要 GIL
纯底层计算       -> 可以考虑释放 GIL
```

## 5. Python 仍然是胶水层

理想结构：

```text
Python:
  参数解析、配置、数据加载、实验组织、错误处理

Extension:
  小而热的核心计算
```

不要把整个应用搬进扩展。那样会失去 Python 的开发效率，同时承担底层复杂度。

## 6. 检查表

写扩展前，按下面问：

1. profiling 是否证明这里是热点？
2. 是否已有库能解决？
3. 是否能通过批量操作减少 Python 循环？
4. 跨语言调用粒度是否足够大？
5. 数据传输成本是否可控？
6. 构建和发布成本是否能接受？
7. Python 和扩展的职责边界是否清楚？

最小心智模型：

1. 扩展是性能关键路径工具。
2. Python 保留调度和工程胶水角色。
3. 扩展边界要少而清楚。
4. 跨语言调用也有成本。
5. 先测量，再下沉。
