# mypy / pyright

mypy 和 pyright 都是静态类型检查器。它们不运行你的程序，而是读取代码和类型标注，推断可能的类型错误。

**类型检查器检查的是代码的静态近似模型；它能提前发现很多接口错误，但不能替代测试、运行时验证和设计判断。**

理解检查器的能力边界，才能用它提升工程质量，而不是和它打架。

## 1. 静态检查做什么

示例：

```python
def add(x: int, y: int) -> int:
    return x + y

add("a", "b")
```

运行时可能返回 `"ab"`，但类型检查器会报：参数类型不匹配。

它检查的是标注表达的约定，而不是 Python 运行时是否能勉强执行。

## 2. mypy 与 pyright 的定位

mypy：

- Python 社区历史较久。
- 配置灵活。
- 很多项目和插件生态成熟。

pyright：

- 速度快。
- VS Code / Pylance 体验好。
- 类型推断和错误提示通常很及时。

两者目标类似，但规则和默认严格程度可能不同。项目里选一个作为准绳，避免团队同时追多个工具的细节差异。

## 3. 配置严格度

类型检查可以渐进开启。不要一开始就追求全项目最严格，尤其是旧代码。

常见策略：

1. 先检查新代码。
2. 给核心模块补标注。
3. 逐步减少 `Any`。
4. 打开更严格选项。
5. 在 CI 中固定检查。

严格度是工程工具，不是荣誉勋章。目标是减少真实错误。

## 4. 常见报错含义

`Optional` 未处理：

```python
user: User | None = find_user()
print(user.name)  # user 可能是 None
```

参数不匹配：

```python
def f(x: int) -> None:
    ...

f("text")
```

返回值不匹配：

```python
def f() -> int:
    return "text"
```

未知类型传播：

```python
from typing import Any

value: Any = load()
value.foo().bar()  # 检查器很难帮你
```

这些错误通常不是“检查器烦”，而是在提醒接口不清楚。

## 5. stub 文件

`.pyi` 文件是类型存根，描述模块接口，不包含实现。

```python
# library.pyi
def parse(text: str) -> object: ...
```

用途：

- 给没有标注的第三方库补类型。
- 给 C 扩展描述接口。
- 分离动态实现和静态接口。

普通项目不一定需要写 `.pyi`，但看到它要知道它是类型接口文件。

## 6. 忽略错误要局部

```python
value = dynamic_api()  # type: ignore[unknown]
```

忽略应该尽量：

- 写在最小范围。
- 附带具体错误码。
- 有注释解释为什么安全。

不要全文件关掉检查，除非是在迁移期，并且有计划收敛。

## 7. 类型检查和测试互补

类型检查擅长发现：

- 参数传错。
- `None` 未处理。
- 返回类型不一致。
- 接口不匹配。
- 重构遗漏。

测试擅长发现：

- 业务逻辑错误。
- 边界条件。
- 外部系统集成问题。
- 性能和副作用。

两者不是替代关系。

## 8. 检查表

使用类型检查器时，按下面问：

1. 项目选择 mypy 还是 pyright 作为准绳？
2. 严格度是否适合当前代码阶段？
3. 报错是在指出真实接口问题，还是工具无法表达动态模式？
4. `Any` 是否正在扩散？
5. `type: ignore` 是否足够局部？
6. 类型检查是否进入 CI？
7. 测试是否覆盖类型检查无法覆盖的行为？

最小心智模型：

1. 类型检查器不运行代码。
2. 它检查静态接口一致性。
3. 严格度可以渐进提高。
4. `Any` 会削弱检查。
5. 类型检查和测试互补。
