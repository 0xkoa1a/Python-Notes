# 渐进式类型系统

Python 的类型系统是渐进式的：你可以从完全不标注开始，逐步给关键边界加约束。

**渐进式类型的目标不是让所有代码立刻变成静态语言，而是在收益最高的位置建立接口边界，并允许动态代码和静态代码共存。**

这很符合 Python 的现实：既要保持表达力，又要让大型工程可维护。

## 1. 什么是渐进式

同一个项目中可以同时存在：

```python
def old_code(x):
    return x.foo()
```

和：

```python
def new_code(user_id: str) -> User:
    ...
```

类型检查器会尽量检查有标注的部分，对未标注或 `Any` 部分保持宽松。

这允许旧项目逐步迁移，而不是一次性重写。

## 2. 先标边界

最高收益的位置：

- 公共函数。
- 模块之间的接口。
- 数据输入输出边界。
- 配置对象。
- 数据库/HTTP/文件系统适配层。
- 被多人调用的工具函数。

低收益的位置：

- 很短的局部变量。
- 临时脚本。
- 明显能从右侧推断的变量。
- 快速探索 notebook。

不要为了标注而标注。

## 3. 限制 `Any` 边界

外部世界常常是动态的：

```python
data = json.loads(text)  # 类型通常很宽
```

策略是：在边界接收动态数据，然后尽快解析成明确类型。

```python
def parse_user(data: object) -> User:
    if not isinstance(data, dict):
        raise TypeError("expected dict")
    ...
    return User(...)
```

让 `Any` 留在边界，不要流进核心业务。

## 4. 静态类型不是运行时验证

```python
def create_user(payload: UserPayload) -> User:
    ...
```

这不代表外部 JSON 自动符合 `UserPayload`。外部输入仍需要运行时验证。

常见分层：

```text
外部 JSON -> 运行时验证/解析 -> 内部类型明确的对象 -> 业务逻辑
```

类型标注保护内部接口，验证保护外部边界。

## 5. 动态特性要集中

Python 支持动态属性、猴子补丁、反射、元编程。它们不是不能用，但要集中在少数边界。

如果整个项目到处都是：

```python
getattr(obj, name)
setattr(obj, name, value)
globals()[name]
```

静态检查器很难帮你。

更好的模式：

- 动态加载集中在插件模块。
- 动态配置集中在解析层。
- 核心业务使用明确类型。

## 6. 好标注是文档

```python
def retry(
    fn: Callable[[], T],
    *,
    attempts: int,
    retry_on: tuple[type[Exception], ...],
) -> T:
    ...
```

这个签名说明了很多事：

- `fn` 不接收参数。
- 返回值和 `fn` 的返回值一致。
- `attempts` 必须关键字传入。
- `retry_on` 是异常类型元组。

好的类型标注让读者少猜。

## 7. 什么时候不标

可以少标或不标：

- 变量类型明显且局部。
- 复杂标注让代码比逻辑还难读。
- 快速实验。
- 动态框架边界，标注成本大于收益。

但公共 API、跨模块边界、复杂数据结构，应该尽量标清楚。

## 8. 迁移策略

实际项目可以这样推进：

1. 给新代码加标注。
2. 给核心模块函数签名加标注。
3. 给配置、DTO、返回对象建模。
4. 引入类型检查器但先宽松。
5. 修掉高价值错误。
6. 限制新增 `Any`。
7. 逐步提高严格度。

类型系统是长期维护工具，不是一周冲刺任务。

## 9. 检查表

推进渐进式类型时，按下面问：

1. 哪些接口最常被调用？
2. 哪些数据来自外部，需要运行时验证？
3. `Any` 是否被限制在边界？
4. 类型标注是否提高了可读性？
5. 是否有复杂标注反而遮蔽了逻辑？
6. 类型检查器是否适度进入 CI？
7. 团队是否知道哪些错误必须修、哪些可以暂缓？

最小心智模型：

1. Python 类型系统是渐进式的。
2. 可以局部标注、逐步增强。
3. 标注优先放在接口边界。
4. `Any` 是动态边界，不应无控制扩散。
5. 静态类型和运行时验证分工不同。
